The VoltSol Sales AI — Strategy & Rollout Plan

Solar Vertical, Phase 1 of a Broader Platform · Drafted 2026-07-03 · Target Launch: January 1, 2027 · Confidential

Written for a mixed audience — some readers here know what a GPU is, some don't, and both need to walk away understanding this plan equally well. Every technical term is defined in plain language the first time it's used, and again in the glossary at the end.

1. The One-Paragraph Version

Right now, when VoltSol talks to a lead, the "brain" behind that conversation is a rented general-purpose AI (Claude, via a service called OpenRouter — more on that below). It's smart, but it's generic: it knows a little about everything, and nothing in particular about how VoltSol actually sells. This plan builds a second brain — one VoltSol owns outright, trained specifically on VoltSol's own sales doctrine, real objection-handling patterns, and both residential and enterprise/commercial deal structures — so the way VoltSol talks to a homeowner or a business owner reflects real closing experience, not a generic script. Solar is the first vertical this gets built and proven on — not the only one. The architecture is designed from day one so that additional verticals can be added later without rebuilding the whole system. Target: live and selling by January 1, 2027.

2. Plain-Language Glossary (read this first if any of Section 3 loses you)

Term What it actually means
AI model A program that has been "shown" huge amounts of text and learned patterns from it — well enough that it can write, answer questions, and hold a conversation. Think of it as a very well-read employee who never sleeps.
LLM (Large Language Model) The technical name for the kind of AI model behind ChatGPT, Claude, etc. "Large" refers to how much it learned from and how big it is internally.
Weights The actual "knowledge" inside a trained AI model — millions or billions of numbers that encode everything it learned.
LoRA adapter The small, separate file our fine-tuning actually produces — not a whole new brain, but a targeted "patch" on top of someone else's base model (see Base model, below). This is the part VoltSol genuinely owns outright and can move freely between hosting providers; the base model underneath stays the property of whoever published it, used under its open license.
Training The process of teaching a model something new by showing it lots of examples, repeatedly, until it changes its behavior. This is expensive and slow, done once (or occasionally), not every time someone asks it a question.
Inference The everyday act of using an already-trained model — asking it a question and getting an answer. This happens every single time VoltSol's AI responds to a lead. Cheap and fast, compared to training.
Frontier model The best, most capable AI models available today (Claude, GPT, Gemini) — run by big companies, rented by the question. VoltSol currently rents Claude for its "brain."
Open-weights model Unlike frontier models (which are locked up inside their maker's servers), some AI models are published publicly — anyone can download the actual weights and run or retrain them. This is what makes owning a custom model possible without being a company the size of Google.
Fine-tuning Taking an open-weights model that already knows how to write and reason, and further training it on a specific skill or voice — in our case, VoltSol's sales methodology — so it behaves differently than the generic version.
LoRA (Low-Rank Adaptation) A cheap, efficient way to fine-tune a model without retraining the whole thing from scratch. Instead of rewriting the entire brain, it adds a small, targeted "patch" of new behavior on top. This is the standard, affordable approach for a project this size — training runs cost roughly $20–200 each, not the millions frontier labs spend.
RAG (Retrieval-Augmented Generation) Instead of baking facts into the model's brain permanently (which goes stale), RAG lets the model look up current facts — prices, tax credit rules, financing terms — from a live, editable database at the moment it answers. Like giving an employee a searchable binder instead of asking them to memorize everything.
Corpus The collected body of text/data used to train or fine-tune a model. Right now, VoltSol's corpus for this specific project is zero — starting from a blank page. Building it is most of this plan.
Base model The starting open-weights model before it's fine-tuned. VoltSol owns zero stake in it, but its open license allows free use and adaptation — the specific model chosen will be whatever is current-generation and best-supported by the hosting provider at the time work actually starts (Section 6), not locked in today.
Serverless A hosting style where you pay only while the model is actively answering a question, instead of renting a computer running 24/7. Cheaper at low volume, but the tradeoff is a possible few-seconds-to-minutes delay ('cold start') if it has to 'wake up' after sitting idle — relevant to Section 7.
Shadow mode A safety step before full launch: the AI drafts real responses to real leads, but nothing sends until a human reviews it. Lets the team see how it performs on real traffic without any customer-facing risk. See Section 9, Phase 4.
GPU (Graphics Processing Unit) The specialized computer chip needed to train and run AI models at reasonable speed. Training rents these by the hour from cloud providers; nobody needs to buy one outright for a project this size.
Quantization A technique to shrink a trained model so it runs on smaller/cheaper hardware with a small accuracy tradeoff. Lets VoltSol run its own model locally instead of always paying a cloud provider per question.
Token The unit AI models read and write text in — roughly ¾ of a word. Costs for renting frontier AI are usually billed per token.
API (Application Programming Interface) The technical doorway that lets one piece of software (VoltSol's system) talk to another (an AI provider, a database, etc.) automatically.
Hallucination When an AI model confidently states something false because it's a next-word-prediction machine, not a database — this is exactly why factual things (prices, tax law) should come from RAG lookups, not the model's memory.
High Availability (HA) Engineering-speak for "this system stays up even if one piece breaks." A single computer with no backup is the opposite of HA.
Uptime The percentage of time a system is actually working and reachable. "99.9% uptime" means roughly 9 hours of downtime per year, tolerated in most business software.
Vertical A distinct business/industry line VoltSol sells into — solar is the first; the term shows up throughout this plan wherever the design deliberately avoids solar-specific hard-wiring.
PPA (Power Purchase Agreement) A financing structure where the customer pays for the solar electricity a system produces, on a per-unit or monthly basis, rather than buying the physical system outright. One of several financing options VoltSol's sales conversations need to handle correctly.
ITC (Investment Tax Credit) A federal tax credit that reduces the cost of installing a solar system by a percentage of its total cost. The rate can increase ("adders") for specific project conditions, such as paying prevailing/union-scale wages on a commercial project.
Legal discovery The legal process where, during a lawsuit, each side can be required to hand over relevant records — including conversation logs. Whoever holds a copy of a record can be legally required to keep those records and hand them over, even if they weren't a party to the original conversation. Relevant to Section 3's case for controlling where conversation data lives.
Data residency A legal/regulatory requirement that certain data must be stored and processed only within specific geographic or organizational boundaries — common in regulated industries, and a factor in why some future verticals may not be able to route data through any convenient third-party AI provider (Section 3).
DPA (Data Processing Agreement) A signed contract that legally binds a vendor to handle a company's data only in specified ways — for example, promising not to retain or train on it. Referenced in Sections 3 and 7 as a hard requirement for any provider touching live conversation data.

3. The Why

The problem with renting a generic brain. Every conversation VoltSol's AI has today runs through Claude via OpenRouter — a capable, general-purpose model that has never sold a solar system, never closed an objection about a $30,000 PPA (power purchase agreement — a contract where the customer pays for the solar power a system produces rather than buying the system outright), and has no institutional memory of what actually works for VoltSol's customers versus what sounds good in theory. It's rented intelligence, not owned expertise, and every dollar spent on it goes to someone else's model with no compounding return to VoltSol.

What's actually valuable and ownable. VoltSol's sales methodology — how to read a homeowner's hesitation, structure a financing conversation, handle "I want to think about it," or negotiate a C&I (commercial & industrial — larger business deals, not just homes) power-purchase structure — is real, hard-won expertise built up over years of closing real deals. Right now, that expertise lives mostly in the heads of a small number of experienced closers and in scattered call recordings. It doesn't scale, it doesn't get better automatically, and it walks out the door if a specific person isn't the one on the call.

The bet: a model fine-tuned specifically on VoltSol's own methodology, backed by a fact database that's always current, can extend that expertise into every conversation VoltSol has — at any hour, on any channel, without diluting quality. To be precise about what "owned" means here: VoltSol owns the fine-tuned adaptation (the LoRA adapter — see glossary) outright and can move it between hosting providers freely. The underlying base model stays the free-to-use, openly-licensed property of whoever published it — VoltSol isn't buying that, doesn't need to, and no one else building on the same open model diminishes what's been built. The real value isn't owning a brain from scratch; it's owning the custom adaptation that encodes VoltSol's actual methodology, with no rented-frontier-model dependency for the routine majority of conversations.

A second, independent reason to own this: control over where sensitive conversation data actually lives. Everything above is about capability and cost. There's a separate, equally real reason to fine-tune an owned model rather than lean permanently on a rented frontier provider (Claude, GPT, Gemini): every message sent to a frontier provider's API passes through that company's infrastructure, under that company's terms of service, subject to that company's data-retention and usage policies — policies VoltSol doesn't control and that can change. Four concrete concerns this creates, worth naming plainly rather than treating as abstract risk:

  • Legal discovery exposure. If VoltSol (or a future vertical's business) is ever party to a lawsuit where sales conversations become relevant evidence, every place a copy of those conversations exists is a place discovery requests can reach — including a third-party AI provider's servers, if that provider retained the traffic. To be precise about what this actually buys VoltSol: VoltSol's own logs and training corpus remain fully discoverable either way — owning the model doesn't make VoltSol's own records disappear from discovery, nor should it. What owning the adapter does remove is the extra, avoidable copy sitting on a third party's servers under that third party's retention policy — and, if a future vertical's requirements justify it, preserves the option of running inference on infrastructure VoltSol fully controls instead of a vendor's, which a permanent frontier-API dependency would foreclose. A signed zero-retention/DPA commitment from any third-party host (Section 7) closes most of this gap even without full self-hosting; owning the adapter is what makes full self-hosting possible if that's ever not enough.
  • IP and methodology protection. The sales doctrine this plan is built on (Section 5) is genuinely valuable, hard-won intellectual property. Every conversation that routes through a third-party frontier API passes through that provider's infrastructure under that provider's terms — terms that, to be fair, the major providers' commercial API agreements typically already commit to not training on by default, and that can include zero-retention arrangements (the same mechanism Section 7 demands from any managed host). The honest framing of the risk isn't that these providers say nothing about data use — it's that those protections exist only as the provider's own revisable contractual terms, not as a structural guarantee, and can be renegotiated or changed unilaterally in a way an owned model's data handling can't be. An owned model shifts the routine majority of conversations off that dependency entirely; it does not eliminate the exposure altogether, since the codified doctrine itself still transits frontier infrastructure elsewhere in this plan by design — the synthetic-data-generation step (Section 5.1) and the failover path (Section 7) both deliberately put the doctrine in front of a frontier model. That's a bounded, accepted cost, not an oversight, and it's covered by the same retention-terms requirement described below, not a separate exposure.
  • Regulatory and contractual data-residency requirements. This doesn't bind solar today, but it matters directly for future verticals (see below): some industries — healthcare, financial services, legal services, government contracting — have explicit rules or contractual obligations about where customer data can be processed and by whom. A platform architecture that depends on routing every conversation through an external frontier API may be blocked outright by some of these requirements, or forced into expensive, vendor-specific compliance offerings on that vendor's own timeline and terms, rather than being blocked categorically in every case. A platform that can run its Behavior and Knowledge layers on infrastructure VoltSol controls avoids being at any single vendor's mercy for that compliance path.
  • Vendor policy and pricing risk. Frontier providers can and do change their terms of service, pricing, rate limits, and model availability unilaterally — a real risk for any business building durable, long-term infrastructure on top of a rented dependency. An owned adapter, portable across hosting providers (Section 7), is structurally insulated from any single vendor's policy shifts in a way a permanently-rented frontier model is not.

This is not a case for abandoning frontier models entirely — the Judgment layer (Section 4) stays rented deliberately, because no small owned model currently matches frontier reasoning for genuinely novel situations, and that tradeoff is worth accepting for the minority of conversations that need it. It is a case for making sure the routine majority of customer-facing conversation, and the underlying methodology that makes VoltSol good at selling, doesn't depend on a third party's infrastructure and policies by default. One tradeoff worth naming plainly, not glossing over: the routing logic in Section 4 sends exactly the highest-stakes conversations — legal/contractual questions, high-value or high-risk situations — to the rented frontier Judgment layer, not away from it. The data-control case above applies most cleanly to the routine majority of traffic; the escalated minority still depends on the frontier provider's own data-handling terms, which is precisely why confirming those terms in writing (Section 7) matters for that path too, not just the primary managed host.

Solar is the first vertical, not the last. This plan is written and built for solar because solar is VoltSol's current business and the fastest place to prove a working sales AI against real leads and real financing conversations already in flight. But the underlying architecture — a Knowledge layer of current facts, a Behavior layer trained on real closing methodology, and a Judgment layer for genuine edge cases (Section 4) — is deliberately built around "how a sales conversation gets won," not hard-wired to solar-specific logic. As VoltSol expands into additional verticals down the road, each new vertical is expected to mean: a new Knowledge/RAG corpus of that vertical's facts, a new (or extended) Behavior-layer corpus of that vertical's objection-handling and closing patterns — while the same Judgment layer, routing logic, and hosting architecture get reused rather than rebuilt from scratch. Solar proves the model works before the same investment is asked to carry a second or third vertical.

Where this fits the bigger picture. This connects directly to the leads-business thesis already in /admin/strategy (see "The Leads-Business Vision"): a proprietary sales AI, trained on real closing methodology across both residential and enterprise deals, is exactly the kind of "recursive learning" moat described there — every VoltSol conversation makes the next one better, in a way a rented frontier model never will, and in a way that compounds further as more verticals feed the same underlying platform.

4. The What

Not one model doing everything — three layers, each doing the job it's actually good at, designed so most of it is reusable across future verticals:

  1. Knowledge layer (RAG — the searchable binder). Facts that change: current pricing tiers, equipment specs, tax credit rules, utility rate schedules, financing program terms, contractor-licensing requirements. Lives in a live, editable database — always current, never "trained in" and then stale. Vertical-specific by design: this is the layer that gets swapped out (not rebuilt) when a new vertical is added — a new fact set, same lookup mechanism.
  2. Behavior layer (the fine-tuned model — the trained sales instinct). How VoltSol actually talks: tone, objection-handling patterns, qualification flow, when to push and when to back off, how a residential pitch differs from a C&I (commercial/industrial) pitch. This is the part that gets fine-tuned on VoltSol's real sales corpus. Partially vertical-specific: the underlying sales instinct (reading hesitation, structuring a close) generalizes; the concrete objection content is retrained per vertical.
  3. Judgment layer (the frontier model — stays rented, on purpose). Genuinely novel situations, edge cases, anything the fine-tuned model hasn't seen before. No small, cheaply-trained model beats a frontier model here, so this plan doesn't try. Claude (via OpenRouter) keeps this job permanently — the plan is never to replace this layer, only to make the Behavior layer good enough that it handles the bulk of routine conversation, with Judgment as the fallback for anything unusual. Fully vertical-agnostic: this layer, the routing logic around it, and the hosting architecture (Section 7) are the parts explicitly built to be reused unchanged as new verticals are added.

How routing actually decides which layer answers — an honest note. It would be convenient if the fine-tuned model could just "know" when it's unsure and hand off automatically. It can't, reliably — small fine-tuned models are not good at accurately judging their own confidence, and treating that as solved would be dishonest. At launch, routing is rule-based, not self-assessed: fixed triggers (specific topics like financing-term specifics, legal/contractual questions, anything the qualification flow flags as high-value or high-risk) send the conversation straight to the Judgment layer or a human, regardless of what the fine-tuned model "thinks." Designing and testing that rule set is itself a deliverable of Phases 4-5 (Section 9), not something that falls out of the fine-tune for free.

What this is not: not a chatbot personality gimmick, not a replacement for a live human closer on complex or high-value calls, not a system that operates without human oversight at launch (see Section 9, Rollout, for the trust-building sequence before this touches real customers unsupervised).

The baseline this has to beat. Before committing further budget past Phase 2, there's a cheaper alternative that has to be tested and ruled out first: once the sales doctrine is codified (Phase 0), simply feeding that doctrine into the existing rented frontier model (Claude) as instructions, paired with the same RAG fact layer, costs almost nothing extra and may already capture most of the value this whole project is chasing. This "prompted-frontier baseline" is now a required comparison arm in the Phase 2 evaluation (Section 6, item 4) — if the fine-tuned model can't beat it, the project stops at Phase 2 rather than continuing to burn budget on a fine-tune that isn't earning its cost.

5. The How — Building the Corpus (this is the actual project)

The honest starting point: VoltSol has zero training data for this today. The base model technology (Section 6) is a solved, well-understood problem. The bottleneck — and therefore the actual multi-month project — is building a real corpus (training data) from scratch. This is "greenfield" work, meaning there is no existing dataset to inherit; every piece of it has to be deliberately built or captured, starting now.

5.1 What we train on, and where it comes from

Source What it captures Status

This is the seed everything else builds on. Not started — first step
Nationwide field-research interviews VoltSol is conducting field interviews with other solar company owners/operators across the country as a deliberate information-gathering effort. Every one of these conversations — different regional objections, different financing norms, different closing styles — is raw training material once recorded and transcribed. This is one of the richest, most unique sources in this entire plan, because it's real-world variation a single-market operation would never otherwise see. In progress — highest priority to capture systematically (see 5.2)
Recorded strategy/sales calls Any recorded conversation involving real sales reasoning, objection handling, or deal structuring — internal calls, partner/industry-contact calls, customer calls (with consent) — is usable raw material once transcribed and cleaned. 1 call captured so far; ongoing as more happen

Not started — needs a dedicated doctrine-writing session focused specifically on VoltSol's enterprise/B2B sales trajectory

5.2 Making the field-research interviews actually useful as training data

This only works as training data if it's captured deliberately, not just remembered anecdotally. Concrete recommendation:

  1. Record every interview (with the other party's consent) — audio at minimum, video if practical.
  2. Transcribe promptly using the same Whisper-based local transcription pipeline already proven on prior recorded calls (openai-whisper — no cloud dependency, no cost).
  3. Tag each transcript with basic metadata: region, company size, residential vs. commercial focus, what was genuinely novel/useful vs. what was already known.
  4. Route through the same human-grading pass as synthetic data before anything gets trained on it — a raw transcript is not automatically good training data; it needs a pass to extract the actual reusable insight from the surrounding conversation.

Recommendation: start this discipline immediately, even before the codified-doctrine step is finished — these interviews are happening in real time regardless of the AI project's timeline, and every uncaptured conversation is a permanently lost data point.

Consent — this is not optional, and recording consent is not the same as training consent. VoltSol representatives are recording other business owners describing their own sales methodology, and this plan proposes using that material to train a commercial product. Agreeing to be recorded is a different thing, legally and ethically, from agreeing to have that recording turned into training data for someone else's AI — and several states, including California (Penal Code §632), require all-party consent to record a conversation at all. Concrete fix, effective before the next interview: the interviewer states on-record, at the start of each interview, something like: "Just so it's on the record — I'm recording this and may use parts of it to help train and improve VoltSol's own sales tools. Cool with you?" No verbal yes on the recording, no use of that interview in the corpus — full stop, no exceptions.

Contingency if field interviews don't deliver. This source is valuable, but the plan cannot depend on it. If these interviews yield little usable material — sparse consent, thin content, or the effort simply doesn't produce enough volume — the codified doctrine plus synthetic bootstrap plus VoltSol's own recorded calls are sufficient on their own to reach a testable v1 fine-tune. Field interviews are enrichment that make the model better across a wider range of regional/market variation; they are not a blocking dependency for Phase 1.

6. The How — Technical Path (once there's a real corpus to train on)

  1. Base model choice: an open-weights model (Section 2 glossary) in the 7-15B parameter range ("B" = billions of internal parameters — bigger generally means smarter but slower/more expensive to run). Deliberately not naming specific models here: open-weights releases move fast, and any model named today will likely be superseded before Phase 2 starts. The real selection criterion (Section 7) is choosing whichever current-generation model is well-supported for cheap serverless hosting by the inference provider selected — that constraint matters more than which specific model wins on paper benchmarks.
  2. Fine-tuning method: LoRA (Section 2) — the efficient, low-cost approach. Typical run costs $20–200 using rented cloud GPU time (via a service like RunPod, using open-source tooling called Axolotl) — there is no need to buy specialized hardware to train this model.
  3. Iteration loop: train → evaluate against real scenarios (does it sound like VoltSol? does it handle objections the way a top VoltSol closer would?) → identify gaps → add more corpus → retrain. Expect multiple rounds, not a single one-shot success.
  4. Evaluation, not vibes: before this ever talks to a real lead, build a fixed test set of at least 50 realistic scenarios, drafted directly from the codified doctrine (ideally during Phase 0, since the doctrine's objection list plus ideal answers is the eval set with almost no extra work). Score every candidate — including the required prompted-frontier baseline described in Section 4 — against the same set, on a simple 1-5 rubric across three dimensions: factual accuracy, voice/tone match, and compliance (does it avoid overpromising, avoid disclosed-bot violations, etc.). Log every version's scores so later fine-tunes can be compared against earlier ones, not just against a feeling. This scored comparison is the gate that decides "ready for shadow mode" (Section 9) — including the go/no-go decision against the frontier baseline.
  5. Integration: the fine-tuned model becomes a specialized skill inside VoltSol's existing OpenClaw agent setup, sitting alongside (not replacing) the frontier-model Judgment layer and the RAG Knowledge layer described in Section 4.
  6. Versioning and rollback — what happens if a fine-tune ships bad. Every trained adapter is kept as a versioned, named artifact, not overwritten. Every new version has to pass the full eval set (item 4) as a regression gate before it's promoted — a new version that scores worse than the current live one on any dimension doesn't ship. The previous version stays hot and ready so a bad rollout can be reverted immediately rather than debugged live. New versions launch to a small percentage of real traffic first, not 100% on day one.

7. Long-Term Hosting & High-Availability Plan

Training vs. serving are two different problems, with two different answers.

Training (the occasional, expensive step of teaching the model something new) always happens on rented cloud GPU time (RunPod, or similar) — this is a solved problem, doesn't need permanent infrastructure, and shouldn't be over-thought. Rent it, run the job, release it.

Serving/inference (the everyday act of the model actually answering real leads) is the part that needs a real availability plan, because by the time this launches, it's touching live customer conversations — the same reliability bar as the rest of VoltSol's stack (which already runs on fully-managed, always-on infrastructure: Vercel, Neon).

  • Local option (cheap, but a single point of failure): VoltSol's existing agent-host hardware (already confirmed capable: Apple M4 Pro chip, 24GB of memory — comfortably able to run a quantized 7-14B fine-tuned model locally via free tools like Ollama or MLX). Attractive because it's already-owned hardware and free to run. But: if that machine's home internet or power goes out, the AI goes dark — unacceptable for a system touching real leads in real time.
  • Recommendation for launch (Jan 1, 2027): don't rely on the home-hosted machine alone for production serving. Use it heavily during the build/test phase (Sections 5-6) where downtime costs nothing. For the actual production launch, pair it with (or fully move to) a managed cloud inference host — services like Fireworks, Together.ai, or a RunPod "serverless" endpoint can host a fine-tuned open-weights model with real uptime guarantees, for a predictable per-use cost, without a single location's home network being a dependency.
  • Recommended production architecture: managed cloud inference as the primary server for the fine-tuned model (real uptime, no home-network dependency), with the local machine kept as a fallback/cost-saver for lower-stakes internal testing and iteration.
  • The real failover — and it's already built. If the fine-tuned endpoint ever goes down, the system falls back to the existing frontier Judgment layer (Claude, with the codified doctrine loaded into its instructions) — the same model VoltSol already runs on today, degrading gracefully to "generic-but-doctrine-aware" rather than going dark. This is the actual high-availability answer, not the local machine: an unexercised local fallback sitting behind home internet isn't real failover. That machine is a dev/test tool, not a production safety net.
  • Data privacy of live conversations — a hard selection criterion, not a nice-to-have. Real customer conversations will flow through whichever inference provider is chosen. Before signing with any provider, confirm in writing: zero-retention inference (they don't store or train on VoltSol's traffic) and a signed Data Processing Agreement (DPA). If a provider won't commit to both, it's disqualified regardless of price. This requirement applies equally to the frontier/OpenRouter path, not just the fine-tuned model's host. The Judgment layer, the failover path, the prompted-frontier baseline, and the synthetic-data-generation step all route conversation data (including the codified doctrine itself) through Claude via OpenRouter today — and per Section 4's routing logic, this is exactly where the highest-stakes, most sensitive conversations end up. Confirming OpenRouter's own logging/retention configuration and the underlying frontier provider's data-use terms in writing is a Phase 3b deliverable (Section 9), to the same standard as the managed host, not an afterthought because that path was already in use before this plan existed.
  • Provider lock-in is genuinely low here — a real strength worth stating plainly. Because the fine-tuned adapter is a small, portable file (Section 2 glossary), VoltSol isn't locked into any one hosting provider the way a fully proprietary system would be. If pricing or reliability sours with one host, the same adapter moves to another provider supporting the same base model, with no retraining required.
  • Post-launch quality monitoring — not just "is it up." An uptime health check catches the server going dark; it does not catch the model quietly getting worse (drifting off-voice, missing new financing terms, etc.). Add a lightweight ongoing check: a weekly human-sampled review of a handful of live transcripts, plus tracking the escalation rate (how often conversations route to the Judgment layer or a human) as a rough quality signal — a rising escalation rate is an early warning, not just an inconvenience.
  • Cost shape to expect — with the real caveat that makes or breaks this number. "Tens of dollars per month" is only realistic if the chosen base model is on the inference provider's supported list for serverless multi-adapter hosting (Fireworks and Together.ai both offer this, but only for base models they explicitly support). If the base model chosen isn't on that list, the alternative is a dedicated GPU endpoint at roughly $1–3/hour — $700–$2,000+/month, a very different number. This is now a hard constraint on model selection (Section 6, item 1): the base model must be chosen from what the target inference provider supports for cheap serverless hosting, not selected on capability alone.
  • Cold-start latency — a real tradeoff for a speed-to-lead product. Serverless endpoints that sit idle between requests can take anywhere from several seconds to a couple of minutes to "wake up" on the next request. At VoltSol's current lead volume, the endpoint will often be cold when a new lead comes in. Mitigation options: pay for an always-warm tier (removes the delay, raises the cost baseline) or accept the occasional slow first response and route time-sensitive first-contact moments through the always-warm frontier Judgment layer instead. This tradeoff should be tested, not assumed away.
  • Full cost picture — line items this plan should track, not just training + hosting. Two real costs are otherwise invisible in this document: (1) frontier-model API usage for synthetic data generation (modest — likely $50–500 depending on volume — but a real line item), and (2) sales leadership's time spent grading corpus batches, which is very likely the single largest true cost of this entire project and deserves to be named as such, not treated as free.
  • Multi-vertical hosting note. Because the Judgment layer, routing logic, and hosting architecture described above are vertical-agnostic by design (Section 4), adding a second vertical later should mean training and deploying an additional adapter/RAG corpus pair on the same infrastructure — not standing up a parallel hosting stack from zero.
  • Data-control tradeoff, stated plainly. The managed-cloud-inference recommendation above is the right default for solar today — it's cheaper and more available than fully self-hosting, and solar carries no data-residency requirement forcing the issue. But it does mean live conversation traffic passes through a third-party host, which only partially addresses the IP/discovery concerns raised in Section 3 (a managed host with a signed DPA and zero-retention terms is a real improvement over a frontier provider's default terms, but it is not the same as VoltSol fully controlling the infrastructure). If a future vertical's legal, contractual, or regulatory requirements demand it, the same owned adapter can be deployed on fully VoltSol-controlled infrastructure instead of a third-party managed host — at the cost of taking on the uptime/HA engineering work a managed host otherwise handles. This is a real option preserved by owning the adapter outright, not a default recommendation for solar's launch.
